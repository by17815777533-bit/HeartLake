# HeartLake 端到端测试与故障排查手册

## 1. 测试目标
交付前确认四类能力同时达标：
- 功能可用：核心路径可完整闭环
- 稳定性：高并发下无明显错误放大
- 安全性：鉴权与风险控制不退化
- 一致性：实时事件和缓存状态可收敛

## 2. 执行顺序（建议）
1. 启动健康检查
2. API 核心链路检查
3. WebSocket 实时链路检查
4. 自动化测试（前端/后端）
5. 综合压测与门槛判定

这样安排的原因：
- 先排基础连通性，再排业务逻辑，再做性能压测，能显著缩短定位路径。

## 4. 关键故障与处理方法

### 4.1 白屏 / 卡在 DDC 脚本加载
表现：
- 控制台停留在 `DDC is about to load ...`

排查步骤：
1. 确认 `3000` 与 `8080` 端口同时在线。
2. 强刷页面清缓存（`Ctrl + Shift + R`）。
3. 检查 Service Worker 是否被旧版本残留干扰。
4. 重启 Flutter web server。

处置原则：
- 优先排“服务未启动”和“缓存脏数据”，不要直接判定业务逻辑异常。

### 4.2 接口 500
表现：
- 如 `/account/privacy`、`/recommendations/similar-stones/*` 返回 500。

排查步骤：
1. 查后端日志的第一条异常栈。
2. 检查数据库迁移是否完整应用。
3. 校验请求体字段与类型。

处置原则：
- 先保证 schema 对齐，再处理业务逻辑。

### 4.3 WebSocket 频繁断连重连
表现：
- 重复出现 disconnected/reconnected 日志。

排查步骤：
1. 检查握手是否带 token。
2. 验证服务端是否接受 query 鉴权或首包鉴权。
3. 检查心跳是否正常回 pong。

处置原则：
- 优先修鉴权链路，再调重连参数。

## 6. 回归建议
- 每次涉及鉴权、缓存、实时链路改动时，至少重跑：
  - `flutter analyze`
  - `flutter test`
  - `ctest -j8`
  - `worldclass_validation.py`

原因：
- 这四项覆盖了“编译质量、功能行为、后端稳定、系统级门槛”四个维度，回归成本最低、收益最高。
