# 技术实现全景手册

> 本文档深入解析 HeartLake 的核心技术实现，覆盖系统架构、算法设计、安全体系及工程决策。适合评审人员快速理解系统的技术深度与创新点。

## 1. 系统架构

### 1.1 DDD 分层架构

后端采用严格的领域驱动设计（DDD）四层架构，通过 `ArchitectureBootstrap` 在启动时按依赖顺序完成初始化：

```
Infrastructure → Domain → Application → EventHandler
```

各层职责：

| 层级 | 目录 | 职责 |
|------|------|------|
| Interfaces | `interfaces/api/` | 20 个 `HttpController`，负责路由注册、请求校验、PASETO v4 认证、响应序列化 |
| Application | `application/` | `ApplicationServiceFactory` 统一创建应用服务；`EventHandlers` 订阅领域事件，编排跨聚合业务流程 |
| Domain | `domain/` | 三个聚合根（Stone / User / Friend），定义仓储接口（`IStoneRepository` / `IUserRepository` / `IFriendRepository`） |
| Infrastructure | `infrastructure/` | 仓储实现、AI 引擎、缓存、事件总线、外部服务集成 |

依赖注入通过 `ServiceLocator` 实现。单例服务使用空删除器包装（`[](T*){}`），避免 DI 容器析构时对静态单例执行 double-free。

### 1.2 事件驱动

`EventBus` 实现发布-订阅模式，解耦领域事件的产生与消费：

| 事件 | 触发时机 | 处理器 | 后续动作 |
|------|----------|--------|----------|
| `StonePublishedEvent` | 石头发布 | `StonePublishedHandler` | AI 情感分析 → 情绪追踪 → 风险评估 |
| `EmotionAnalyzedEvent` | 情感分析完成 | `EmotionAnalyzedHandler` | 缓存失效 → 推荐引擎刷新 |
| `RippleCreatedEvent` | 涟漪互动 | `RippleCreatedHandler` | 缓存失效 → 实时广播协同 |
| `BoatSentEvent` | 纸船发送 | `BoatSentHandler` | 纸船审计日志 → 统一评论链路 |

### 1.3 两级缓存体系

```
请求 → CacheManager（L1 进程内） → RedisCache（L2 分布式） → PostgreSQL
```

L1 `CacheManager`：
- 数据结构：LRU 双向链表 + `unordered_map` 索引，O(1) 读写
- 并发控制：`shared_mutex` 读写锁（读共享、写独占）
- 淘汰策略：惰性 TTL 过期 + 容量驱逐
- 默认配置：容量 10000 条，TTL 300 秒

L2 `RedisCache`：
- 分布式共享缓存，适用于多实例部署场景

一致性保障：写操作成功后按前缀主动失效相关缓存（`invalidatePattern`），保证最终一致性。

### 1.4 跨端集合契约治理

为了避免管理端、后端、移动端分别维护各自的分页/列表字段拼装逻辑，项目将集合响应契约收敛为“三端单一来源”：

| 层级 | 单一入口 | 职责 |
|------|----------|------|
| 后端 | `backend/include/utils/ResponseUtil.h` | `buildPaginationPayload()` 与 `buildCollectionPayload()` 统一输出语义主键、`items/list` 别名和分页元数据 |
| 移动端 | `frontend/lib/data/datasources/social_payload_normalizer.dart` | `extractNormalizedList()`、`extractPaginationPayload()`、`buildCollectionEnvelope()` 统一读取并镜像后端集合载荷，`user_service.dart`、`account_service.dart` 与 `recommendation_response_parser.dart` 也复用这条能力链 |
| 管理端 | `admin/src/utils/collectionPayload.ts`、`admin/src/composables/useDashboardLoaders.ts` | 管理台列表页与 Dashboard loader 按同一契约解析 `data`、语义键、`items/list` 与分页信息 |

这样做的工程收益：

- 控制器和服务层不再重复手写 `total/page/page_size/has_more/pagination`，降低字段漂移概率
- 前端页面不再直接耦合某个单独接口的临时字段命名，兼容 `boats/items/list`、`notifications/items/list` 等多种形态
- 通用集合归一化器额外兼容 `count/total_count/meta.total/hasMore` 以及 `data.items`、`messages.items` 这类嵌套列表，避免“接口成功但分页被静默压扁成单页”
- 账号安全链中的 `devices/logs/events/blocked_users` 以及“收到的纸船”也走同一套集合壳，客户端不会再因为某个接口只返回语义键或只返回 `items/list` 而丢分页元数据
- 认证恢复链会持久化 `session_id`，这样登录设备、刷新令牌和客户端会话状态终于能围绕同一条会话主键协同，而不是只靠 `user_id` 做弱关联
- 互动链服务层也补上了“空页 total fallback”：即使翻到超尾页，通知、纸船、涟漪列表仍会返回真实 `total/unread_count`，不会错误回落为 0
- `paper_boats` 的计数维护不再只依赖主评论链：`sendBoat()` 旁路写入、后台软删纸船、永久删号都同步回补 `stones.boat_count`，避免资料页和湖神扫描继续吃到脏计数
- 管理台 `Dashboard` 不再为 `topics/hours/stats/trends` 各自维护一套“数组还是对象”的兼容分支，`useDashboardLoaders` 统一走集合归一化
- 文档、测试、运行时返回结构可以围绕同一份契约描述，便于论文中解释“跨端一致性如何保证”

当前已经收敛到这条出口的后端链路包括：

- `InteractionApplicationService`、`PaperBoatController`
- `StoneApplicationService`、`UserApplicationService`
- `UserController`、`ReportController`、`RecommendationController`、`AccountController`、`TempFriendController`、`FriendController`、`ConsultationController`
- `AdminController`、`AdminManagementController`

### 1.5 移动端 Provider 依赖反转

为了避免状态层直接绑定单例实现、导致真实 Provider 难以验证且难以替换，移动端进一步把 Provider 依赖统一收敛为接口注入：

| 层级 | 单一入口 | 职责 |
|------|----------|------|
| 数据源接口 | `frontend/lib/data/datasources/notification_service.dart` 等 | 暴露 `NotificationDataSource`、`FriendDataSource`、`TempFriendDataSource`、`StoneDataSource`、`AuthDataSource`、`InteractionDataSource` 等最小能力接口 |
| 实时通道接口 | `frontend/lib/data/datasources/websocket_manager.dart` | 通过 `WebSocketClient` / `RealtimeClient` / `RoomSubscriptionClient` 抽象统一事件订阅、连接、断开与房间订阅契约 |
| 缓存接口 | `frontend/lib/data/datasources/cache_service.dart` | 通过 `CacheStore` 抽象分页缓存读写和前缀失效 |
| 状态层 | `frontend/lib/presentation/providers/*.dart` | Provider 只依赖抽象接口，默认走 `service_locator`，测试和未来替换实现时可直接注入 |

这样做的工程收益：

- Provider 从“单例直连”变成“抽象注入”，生产环境和测试环境复用同一条执行路径
- WebSocket 监听器可精确注册、精确解绑，状态层生命周期更可控
- 后续若要接入缓存层、离线数据源或 mock 网关，不需要改 Provider 业务逻辑
- `UserProvider` 不再直接读取 `StorageUtil` / `ApiClient`，会话恢复与资料拉取统一收敛到 `AuthDataSource`
- `AuthService` 进一步向外暴露认证事件流，`UserProvider` 在登录 / 恢复 / 刷新 / 退出时会被自动推送最新会话，WebSocket 建链不再依赖页面自己记得同步状态
- 匿名登录 / 恢复登录 / `POST /api/auth/refresh` 现在统一走服务端 `user_sessions` 会话链：登录即下发 `refresh_token`，冷启动可直接续期访问令牌，旧客户端携带 Bearer 访问 `refresh` 时也会自动回补会话记录
- `StoneProvider` 不再只缓存列表本身，而是缓存完整集合载荷，缓存命中时也能恢复 `has_more`、分页元数据和当前页状态
- `StoneProvider` 进一步接管湖面实时房间订阅与湖面气象数据，`LakeScreen` / `LakeFeedScreen` 不再各自维护一套分页、WebSocket 监听和本地索引
- `StoneProvider` 在重连后只会于真实湖面订阅存在时重新入房并刷新天气/列表，避免空页面状态下的无效重放
- `WebSocketManager` 通过 `RoomSubscriptionClient` 暴露房间级契约，页面和 Provider 之间的职责边界从“直接操作单例”收敛为“由 Provider 托管实时订阅”
- `StoneCardController` 不再直接硬编码 `InteractionService` / `WebSocketManager` / `StorageUtil`，而是经由抽象依赖组装，石头卡片层的交互逻辑开始具备可替换能力
- `InteractionDataSource` 继续向“交互唯一入口”收敛，`stone_detail`、`my_boats`、`my_ripples` 相关读取与删除流程不再由页面直接拼请求和本地假状态
- 互动列表页的本地移除动作改为以后端成功结果为准，页面只消费 Provider / 数据源给出的稳定状态，避免“界面删了但后端没成”的假同步
- `FriendsScreen`、`TempFriendsScreen` 不再各自维护一套好友/临时好友列表和实时监听逻辑，页面状态统一回收到 `FriendProvider`

## 2. EdgeAI 引擎

`EdgeAIEngine` 是系统 AI 能力的核心，采用门面模式统一管理 8 个子系统。初始化采用“首次构建子系统 + 后续运行时配置热应用”模式：首次 `initialize()` 只负责构建实例，之后后台配置变更会同步下推到 HNSW、差分隐私、情绪脉搏和本地情感缓存，不再停留在配置文件层面。当前启动链已经不再把 `edge_ai_enabled` 硬编码为开启，后台管理端持久化的 `enabled / onnx_enabled / model_path / vocab_path / onnx_threads` 会在重启后真实参与引擎初始化，而不是只写进配置文件；当 EdgeAI 以禁用态启动时，核心子系统也会保持延迟初始化，避免 2c2g 节点为一条关闭的 AI 能力链提前分配无谓内存和构建成本。

### 2.1 情感分析 — SentimentAnalyzer

三层融合分析流水线：

1. 规则层（`ruleSentiment`）：emoji 和标点符号模式匹配，快速捕捉明确的情感信号
2. 词典层（`lexiconSentiment`）：中英文情感词典匹配，支持程度副词（`intensifiers_`）和否定词（`negators_`）修饰
3. 统计层（`statisticalSentiment`）：词汇多样性（TTR）、词长分布等文本统计特征

输出 `EdgeSentimentResult`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `score` | `float [-1.0, 1.0]` | 情感分数 |
| `mood` | `string` | 情绪类型（happy / sad / angry / anxious / neutral 等） |
| `confidence` | `float [0.0, 1.0]` | 置信度 |

性能优化：
- LRU 缓存（默认 4096 条，TTL 300 秒），`shared_mutex` 保护
- 飞行中请求去重（`sentimentInFlight_`）：相同文本的并发请求共享同一个 `shared_future`，避免重复计算
- 可选 ONNX Runtime 神经网络推理（`#ifdef HEARTLAKE_USE_ONNX`）
- `EdgeAIEngine` 会在 2c2g 级别节点自动切到低资源默认值：压低 `HNSW efConstruction/efSearch`、缩小本地情感缓存上限，并把 ONNX 默认线程数收敛到 `1`
- admin 端 EdgeAI 配置接口现在同时返回 flat root 字段和 `config` 镜像，并明确区分“运行时立即生效字段”和“需要重启生效字段”；`enabled`、`hnsw_*`、`dp_*`、`pulse_window_seconds`、`sentiment_cache_ttl` 会立即热应用，`onnx_enabled`、模型路径、词表路径和 ONNX 线程数则只持久化并要求重启
- `vectorSearch` 返回 HNSW 相似度结果时，会按命中的 `stone_id` 批量回填 `content / mood_type / created_at`，admin 检索面板不再只能看到裸 ID

### 2.2 内容审核 — ContentModerator

两阶段审核流程：

阶段一 — AC 自动机多模式匹配（O(n) 复杂度）：
- 四大类别：self_harm（自伤）、violence（暴力）、sexual（不当内容）、profanity（脏话）
- `buildModerationAC()` 构建 Aho-Corasick 自动机

阶段二 — 五因子心理风险评估（`FiveFactorRiskDetail`）：

| 因子 | 权重 | 检测内容 |
|------|------|----------|
| 自伤意图 | 0.9 | 自伤相关关键词与表达模式 |
| 绝望表达 | 0.6 | 绝望、无助类语言表达 |
| 时间紧迫性 | 0.5 | "最后一次"等紧迫性表达 |
| 语言风险标记 | 0.3 | 高风险语言模式匹配 |
| 社交孤立 | 0.1 | 孤立感相关表达 |

输出 `EdgeModerationResult`：风险等级（safe / low_risk / medium_risk / high_risk）、匹配模式列表、是否需要紧急关注（`needsAlert`）。

### 2.3 HNSW 向量检索 — HNSWIndex

Hierarchical Navigable Small World 图索引，提供高性能近似最近邻搜索。

核心参数：

| 参数 | 值 | 说明 |
|------|-----|------|
| `M` | 16 | 每层最大邻居数 |
| `mMax0` | 32 | 第 0 层最大邻居数 |
| `efConstruction` | 200 | 构建阶段搜索宽度 |
| `efSearch` | 50 | 查询阶段搜索宽度 |

算法增强：

- Ada-EF 自适应搜索宽度：根据查询难度动态调整 ef 参数，Hub Highway 机制优化长尾路由
- Matryoshka 融合 cosine 重排序（`rerankCandidates`）：对粗排结果执行精确重排
- RND 邻居多样性选择（`selectDiverseNeighbors`）：角度多样性剪枝——若候选 c 到某已选邻居 n 的距离小于 c 到基准点的距离，则 n 已覆盖 c 的方向，跳过 c
- ADSampling 提前终止距离计算（`vectorDistanceThreshold`）：部分和超过阈值时提前返回 +inf
- `thread_local` visited 标记：避免 `shared_lock` 下的数据竞争

并发控制：`shared_mutex` 读写锁，搜索操作持共享锁，插入 / 删除持独占锁。

2026-03 的工程收敛点：
- 搜索结果携带内部 `nodeIndex`，二阶段重排不再重复回查 `idMap_`
- 删除向量后会重建入口点与 `maxLevel_`，避免逻辑删除节点继续污染入口
- 管理端重建 embedding 时同步刷新内存 HNSW 索引，插入路径改成 upsert 语义
- 旧 embedding 回退链优先复用 `stone_embeddings` 中已持久化的源向量，并把候选保留限制在 top-K，避免全量收集排序

### 2.4 差分隐私 — EdgeDifferentialPrivacy

两种噪声机制：

Laplace 机制：
- 噪声分布：`noise ~ Lap(Δf / ε)`
- 适用场景：标量和低维向量
- 向量场景按组合定理均分 ε 到每维

Gaussian 机制：
- 噪声标准差：`σ = Δ₂ · √(2·ln(1.25/δ)) / ε`
- 适用场景：高维向量，噪声尺度 O(√d) 优于 Laplace 的 O(d)
- d=128 时噪声降低约 11 倍

隐私预算追踪采用 zCDP（zero-Concentrated DP）框架：
- Gaussian 操作消耗 ρ = Δ²/(2σ²)，ρ 线性可加
- 通过 ρ → (ε, δ) 转换公式回推等效消耗
- 同时维护 ε / δ / ρ 三个 `std::atomic<float>` 计数器，支持混合机制场景

### 2.5 联邦学习 — FederatedLearner

借鉴 FedProx 处理端侧计算性能差异，提供隐私保护的个性化情感模型服务。

FedAvg 联邦平均聚合算法：

```
w_global = Σ(n_k / N) · clip(w_k, C) + N(0, σ²)
```

| 符号 | 含义 |
|------|------|
| `n_k` | 节点 k 的样本数 |
| `N` | 总样本数 |
| `C` | L2 梯度裁剪阈值（防止单节点过度影响全局模型） |
| `σ` | DP 高斯噪声标准差（保护隐私） |

支持 FedProx 近端正则化（`mu` 参数），约束本地模型不偏离全局模型过远。每轮聚合后清空本地模型缓冲区，轮次计数器递增。

### 2.6 情绪脉搏 — EmotionPulseDetector

基于滑动窗口的实时情绪统计：

- 时间衰减加权：指数衰减函数，越新的样本权重越高
- 置信度加权：低置信度样本降权
- EWMA 平滑趋势斜率
- 鲁棒抗噪：中位数 + MAD（Median Absolute Deviation）离群点检测

输出 `EmotionPulse` 快照：加权平均分、标准差、情绪分布、趋势斜率、主导情绪。

### 2.7 模型量化 — ModelQuantizer

基于整数对称量化实现 float32 → INT8 推理：

- `quantizeToInt8`：计算 `scale = max(|tensor|) / 127`，`zeroPoint = 0`
- `quantizedMatMul`：INT8 矩阵乘法，手动 4 路循环展开优化
- `quantizedForward`：量化输入 → 矩阵乘 → 加偏置 → ReLU 激活

### 2.8 节点监控 — EdgeNodeMonitor

边缘节点健康监控：节点注册、状态上报（CPU / 内存 / 延迟）、负载均衡选择最优节点。

## 3. 情绪共鸣推荐

### 3.1 EmotionResonanceEngine

四维共鸣评分算法：

```
ResonanceScore = α·SemanticSim + β·TrajectoryDTW + γ·TemporalDecay + δ·DiversityBonus
```

默认权重及设计依据：

| 维度 | 权重 | 依据 |
|------|------|------|
| β 情绪轨迹匹配 | 0.35 | 核心差异化特性，相似情绪旅程的用户互动率最高 |
| α 语义相似度 | 0.30 | 保证内容相关性 |
| γ 时间衰减 | 0.20 | 避免推荐过旧内容 |
| δ 多样性奖励 | 0.15 | 防止回音室效应 |

DTW 优化：
- LB_Keogh 下界：O(n) 快速下界估计，用于剪枝
- LB_Improved 双向包络下界：比 LB_Keogh 更紧，剪枝率提升 20-40%
- Early Abandoning DTW：部分累积超过 best-so-far 时提前终止

权重自适应：
- EMA（指数移动平均）在线学习，根据用户隐式反馈调整四维权重
- `atomic<shared_ptr<const ResonanceWeights>>` 无锁一致读取，杜绝 torn-read

2026-03 的推荐链改造：
- 共鸣推荐路径把候选作者近 7 天轨迹改成单次批量聚合查询，移除了控制器逐条查 stone 详情的 N+1
- 候选集在 SQL 阶段就排除当前用户已经互动过的 stone，并统一补齐 `content / mood_type / emotion_score / author / ripple_count / created_at`
- 情绪趋势接口收敛为“按天唯一聚合 + 主导 mood 选择”，避免同一天返回多条趋势点导致前端时序抖动
- 搜索推荐接口改成窗口总数分页，普通页不再额外做一次 `COUNT(*)` 查询，只在空页回退时补查总数

2026-03 热路径补充优化：
- `findResonance` 直接在 SQL 中预计算 `temporal_score`，热路径不再为每个候选反复解析时间字符串
- 候选评分循环改成单次权重快照，避免每条候选都做原子读取；`results / recommendedMoods / trajectoryCache` 也按候选规模预留容量
- 下界剪枝收敛为 `LB_Improved` 单路径，避免同一对轨迹重复跑 `LB_Keogh + LB_Improved`
- `LB_Keogh / LB_Improved` 的包络计算改成 deque 滑窗，真正落到 O(n)；`trajectorySimDTW_EA` 也修正为按归一化距离阈值 early-abandon
- `ResonanceSearchService` 的 Milvus/DB 候选重排也已收口为批量元数据回填 + 批量轨迹加载，不再为每个候选单独回查石头和情绪轨迹，避免共鸣搜索重新退化成 N+1

### 3.2 RecommendationEngine

五种推荐策略融合：

| 策略 | 方法 |
|------|------|
| User-CF | 基于潜在因子余弦相似度的用户协同过滤 |
| Item-CF | 基于共现矩阵的物品协同过滤 |

| 图传播 | 简化 PageRank 在用户-物品二部图上的传播 |

在线学习机制：
- SGD 增量更新潜在因子（`updateLatentFactors`）
- Thompson Sampling 自适应探索率
- 时间衰减权重：指数半衰期 24 小时

MMR（Maximal Marginal Relevance）多样性重排序，避免推荐结果过于同质。推荐结果元数据也统一稳定输出 `content`、`mood_type`、`emotion_score`、`author_id`、`author_name`、`ripple_count`、`created_at`，方便三端共用解析链。

2026-03 补充：

- `hybridRecommend` 内部三路候选（User-CF / Item-CF / 内容推荐）已改为复用同一个 DB client，并统一走单一 `RecommendationCandidate` 构造路径；边缘节点不再为同一轮推荐重复创建连接、重复拼装 `content/mood_type/author/ripple_count` 元数据

### 3.3 AdvancedEmbeddingEngine

| 特征类型 | 维度占比 | 方法 |
|----------|----------|------|

| 情感特征 | 15 维 | 中英文情感词典，提取极性 / 强度 / 波动 / 复杂度 |
| 统计特征 | 15 维 | 词汇多样性（TTR）、词频熵、词长分布 |

性能优化：
- LRU 缓存（默认 10000 条）

- N-gram 内联滑窗哈希，零中间 string 分配
- `std::jthread` 后台异步预热训练语料

## 4. 语义缓存

`SemanticCache` 实现两级缓存架构：

- L1 精确匹配：SHA256 哈希 → `unordered_map`，O(1)
- L2 语义匹配：查询 embedding 插入 `HNSWIndex`，O(log n) ANN 检索

当最近邻相似度超过阈值（默认 0.92）时视为语义命中，复用已缓存的响应。

淘汰策略：LFU（最少使用频率优先）+ TTL 过期清理（默认 86400 秒）。

预期效果：缓存命中率从 30% 提升至 60-70%，减少 40-50% AI API 调用。

## 5. 双记忆 RAG

`DualMemoryRAG` 双记忆架构：

短期记忆（`ShortTermEntry`）：
- 保留最近 5 次交互上下文
- 记录内容、情绪、分数、时间戳、访问次数和预计算 embedding
- 基于相关性淘汰（`evictLeastRelevant`），而非简单 FIFO

长期记忆（`LongTermProfile`）：
- 平均情绪分、指数衰减加权分（Ebbinghaus 遗忘曲线，λ=0.05）
- 主导情绪、情绪波动度、情绪趋势（rising / falling / stable）
- 连续负面天数（用于风险预警）

隐私保护：所有记忆使用 shadow_id，不关联真实身份。

`buildRAGPrompt` 将双记忆上下文注入 AI 提示词，生成个性化的情感回复。

2026-03 的性能治理：
- 新写入短期记忆时同步缓存 embedding，相关检索和淘汰优先复用已缓存向量
- 被命中的短期记忆会真实更新 `accessCount / lastAccessTime`，淘汰策略不再依赖失效统计字段
- 长期画像改成最短 180 秒刷新一次，且数据库聚合在锁外执行，避免每次对话都持锁刷库
- `getEmotionInsights` 共享同一套长期画像节流策略，并显式输出 `decay_weighted_score`
- 长期画像聚合已进一步合并成单次 CTE 查询，并通过 `refreshInFlight` 去重并发刷新，避免同一用户在高并发下重复击穿数据库
- `generateResponse` 复用当前轮 `embedding` 完成短期写入、相关检索和淘汰，减少一次热路径重复向量生成；时间戳格式化也改成线程安全实现

## 6. 安全体系

### 6.1 PASETO v4 认证

替代 JWT 的无状态令牌方案，从根本上避免 JWT 的算法混淆攻击（`alg: none`）。`SecurityAuditFilter` 在请求到达 Controller 前统一校验，提取 `user_id` 注入请求属性。

管理后台使用独立的 `AdminAuthFilter` 和独立签名密钥（`ADMIN_PASETO_KEY`），与用户端完全隔离。

### 6.2 E2E 端到端加密

`E2EEncryption` 实现 IND-CCA2 安全性：

| 环节 | 算法 |
|------|------|
| 密钥交换 | X25519 ECDH（`computeSharedSecret`） |
| 密钥派生 | HKDF-SHA256（`deriveSessionKey`） |
| 认证加密 | AES-256-GCM（96-bit IV, 128-bit tag） |
| 前向保密 | 临时密钥对，每次会话独立 |

### 6.3 内容安全过滤

`ContentFilter` 三级过滤架构：

1. `CountingBloomFilter` 预检测：快速排除明显安全的文本，减少后续计算开销
2. `ACAutomaton` 精确匹配：Aho-Corasick 多模式匹配，O(n) 复杂度
3. `ShardedLRUCache` 结果缓存：热点文本直接命中

支持运行时热更新（`reload`），词条按类别和级别分级，白名单机制减少误报。文本预处理（`normalize`）统一全半角、移除零宽字符等干扰。

### 6.4 熔断器

`CircuitBreaker` 三态状态机：

```
CLOSED ──(连续失败达阈值)──→ OPEN ──(冷却超时)──→ HALF_OPEN
   ↑                                                    │
   └──────────(试探成功)────────────────────────────────┘
                                    │
                              (试探失败)
                                    ↓
                                  OPEN
```

`execute()` 采用两阶段锁策略：持锁检查状态（Phase 1），释放锁后执行业务逻辑（Phase 2），避免长时间持锁阻塞其他调用者。

## 8. 高性能工程实践

### 8.1 并发安全

| 机制 | 应用场景 |
|------|----------|
| `std::shared_mutex` | 读多写少场景（缓存、HNSW 索引、情绪脉搏） |
| `std::atomic` | 无锁计数器（隐私预算、统计指标、启用标志） |
| `std::atomic<shared_ptr>` | 无锁一致读取（共鸣权重） |
| `std::once_flag` | 单例初始化 |
| `thread_local` | HNSW visited 标记，避免共享锁下的数据竞争 |

### 8.2 内存管理

- `std::unique_ptr` 管理子系统生命周期
- 空删除器包装静态单例，避免 DI 容器 double-free
- LRU 缓存容量限制 + TTL 过期，防止内存无限增长

### 8.3 算法优化

- 4 路循环展开：向量距离计算、归一化、余弦相似度
- ADSampling 提前终止：距离计算部分和超过阈值时返回 +inf
- 特征哈希降维：避免高维稀疏向量的内存开销
- N-gram 内联滑窗哈希：零中间 string 分配
