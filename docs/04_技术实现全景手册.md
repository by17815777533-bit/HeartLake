# 技术实现全景手册

> 本文档深入解析 HeartLake 的核心技术实现，覆盖系统架构、算法设计、安全体系及工程决策。适合评审人员快速理解系统的技术深度与创新点。

## 1. 系统架构

### 1.1 DDD 分层架构

后端采用严格的领域驱动设计（DDD）四层架构，通过 `ArchitectureBootstrap` 在启动时按依赖顺序完成初始化：

```
Infrastructure → Domain → Application → EventHandler
```

各层职责：

| 层级 | 目录 | 职责 |
|------|------|------|
| Interfaces | `interfaces/api/` | 20 个 `HttpController`，负责路由注册、请求校验、PASETO v4 认证、响应序列化 |
| Application | `application/` | `ApplicationServiceFactory` 统一创建应用服务；`EventHandlers` 订阅领域事件，编排跨聚合业务流程 |
| Domain | `domain/` | 三个聚合根（Stone / User / Friend），定义仓储接口（`IStoneRepository` / `IUserRepository` / `IFriendRepository`） |
| Infrastructure | `infrastructure/` | 仓储实现、AI 引擎、缓存、事件总线、外部服务集成 |

依赖注入通过 `ServiceLocator` 实现。单例服务使用空删除器包装（`[](T*){}`），避免 DI 容器析构时对静态单例执行 double-free。

### 1.2 事件驱动

`EventBus` 实现发布-订阅模式，解耦领域事件的产生与消费：

| 事件 | 触发时机 | 处理器 | 后续动作 |
|------|----------|--------|----------|
| `StonePublishedEvent` | 石头发布 | `StonePublishedHandler` | AI 情感分析 → 情绪追踪 → 风险评估 |
| `EmotionAnalyzedEvent` | 情感分析完成 | `EmotionAnalyzedHandler` | 缓存失效 → 推荐引擎刷新 |
| `RippleCreatedEvent` | 涟漪互动 | `RippleCreatedHandler` | 通知推送 → 亲密度计算 |
| `BoatSentEvent` | 纸船发送 | `BoatSentHandler` | 匿名消息投递 |

### 1.3 两级缓存体系

```
请求 → CacheManager（L1 进程内） → RedisCache（L2 分布式） → PostgreSQL
```

L1 `CacheManager`：
- 数据结构：LRU 双向链表 + `unordered_map` 索引，O(1) 读写
- 并发控制：`shared_mutex` 读写锁（读共享、写独占）
- 淘汰策略：惰性 TTL 过期 + 容量驱逐
- 默认配置：容量 10000 条，TTL 300 秒

L2 `RedisCache`：
- 分布式共享缓存，适用于多实例部署场景

一致性保障：写操作成功后按前缀主动失效相关缓存（`invalidatePattern`），保证最终一致性。

### 1.4 跨端集合契约治理

为了避免管理端、后端、移动端分别维护各自的分页/列表字段拼装逻辑，项目将集合响应契约收敛为“三端单一来源”：

| 层级 | 单一入口 | 职责 |
|------|----------|------|
| 后端 | `backend/include/utils/ResponseUtil.h` | `buildPaginationPayload()` 与 `buildCollectionPayload()` 统一输出语义主键、`items/list` 别名和分页元数据 |
| 移动端 | `frontend/lib/data/datasources/social_payload_normalizer.dart` | `extractNormalizedList()`、`extractPaginationPayload()`、`buildCollectionEnvelope()` 统一读取并镜像后端集合载荷，`user_service.dart` 与 `recommendation_response_parser.dart` 也复用这条能力链 |
| 管理端 | `admin/src/utils/collectionPayload.ts` | 管理台页面按同一契约解析 `items/list`、语义键与分页信息 |

这样做的工程收益：

- 控制器和服务层不再重复手写 `total/page/page_size/has_more/pagination`，降低字段漂移概率
- 前端页面不再直接耦合某个单独接口的临时字段命名，兼容 `boats/items/list`、`notifications/items/list` 等多种形态
- 文档、测试、运行时返回结构可以围绕同一份契约描述，便于论文中解释“跨端一致性如何保证”

典型集合响应结构如下：

```json
{
  "boats": [{ "...": "..." }],
  "items": [{ "...": "..." }],
  "list": [{ "...": "..." }],
  "total": 42,
  "page": 2,
  "page_size": 10,
  "pageSize": 10,
  "total_pages": 5,
  "totalPages": 5,
  "has_more": true,
  "pagination": {
    "total": 42,
    "page": 2,
    "page_size": 10,
    "pageSize": 10,
    "total_pages": 5,
    "totalPages": 5,
    "has_more": true
  }
}
```

## 2. EdgeAI 引擎

`EdgeAIEngine` 是系统 AI 能力的核心，采用门面模式统一管理 8 个子系统。单例 + `std::once_flag` 保证线程安全的一次性初始化。

### 2.1 情感分析 — SentimentAnalyzer

三层融合分析流水线：

1. 规则层（`ruleSentiment`）：emoji 和标点符号模式匹配，快速捕捉明确的情感信号
2. 词典层（`lexiconSentiment`）：中英文情感词典匹配，支持程度副词（`intensifiers_`）和否定词（`negators_`）修饰
3. 统计层（`statisticalSentiment`）：词汇多样性（TTR）、词长分布等文本统计特征

输出 `EdgeSentimentResult`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `score` | `float [-1.0, 1.0]` | 情感分数 |
| `mood` | `string` | 情绪类型（happy / sad / angry / anxious / neutral 等） |
| `confidence` | `float [0.0, 1.0]` | 置信度 |

性能优化：
- LRU 缓存（默认 4096 条，TTL 300 秒），`shared_mutex` 保护
- 飞行中请求去重（`sentimentInFlight_`）：相同文本的并发请求共享同一个 `shared_future`，避免重复计算
- 可选 ONNX Runtime 神经网络推理（`#ifdef HEARTLAKE_USE_ONNX`）

### 2.2 内容审核 — ContentModerator

两阶段审核流程：

阶段一 — AC 自动机多模式匹配（O(n) 复杂度）：
- 四大类别：self_harm（自伤）、violence（暴力）、sexual（不当内容）、profanity（脏话）
- `buildModerationAC()` 构建 Aho-Corasick 自动机

阶段二 — 五因子心理风险评估（`FiveFactorRiskDetail`）：

| 因子 | 权重 | 检测内容 |
|------|------|----------|
| 自伤意图 | 0.9 | 自伤相关关键词与表达模式 |
| 绝望表达 | 0.6 | 绝望、无助类语言表达 |
| 时间紧迫性 | 0.5 | "最后一次"等紧迫性表达 |
| 语言风险标记 | 0.3 | 高风险语言模式匹配 |
| 社交孤立 | 0.1 | 孤立感相关表达 |

输出 `EdgeModerationResult`：风险等级（safe / low_risk / medium_risk / high_risk）、匹配模式列表、是否需要紧急关注（`needsAlert`）。

### 2.3 HNSW 向量检索 — HNSWIndex

Hierarchical Navigable Small World 图索引，提供高性能近似最近邻搜索。

核心参数：

| 参数 | 值 | 说明 |
|------|-----|------|
| `M` | 16 | 每层最大邻居数 |
| `mMax0` | 32 | 第 0 层最大邻居数 |
| `efConstruction` | 200 | 构建阶段搜索宽度 |
| `efSearch` | 50 | 查询阶段搜索宽度 |

算法增强：

- Ada-EF 自适应搜索宽度：根据查询难度动态调整 ef 参数，Hub Highway 机制优化长尾路由
- Matryoshka 融合 cosine 重排序（`rerankCandidates`）：对粗排结果执行精确重排
- RND 邻居多样性选择（`selectDiverseNeighbors`）：角度多样性剪枝——若候选 c 到某已选邻居 n 的距离小于 c 到基准点的距离，则 n 已覆盖 c 的方向，跳过 c
- ADSampling 提前终止距离计算（`vectorDistanceThreshold`）：部分和超过阈值时提前返回 +inf
- `thread_local` visited 标记：避免 `shared_lock` 下的数据竞争

并发控制：`shared_mutex` 读写锁，搜索操作持共享锁，插入 / 删除持独占锁。

### 2.4 差分隐私 — EdgeDifferentialPrivacy

两种噪声机制：

Laplace 机制：
- 噪声分布：`noise ~ Lap(Δf / ε)`
- 适用场景：标量和低维向量
- 向量场景按组合定理均分 ε 到每维

Gaussian 机制：
- 噪声标准差：`σ = Δ₂ · √(2·ln(1.25/δ)) / ε`
- 适用场景：高维向量，噪声尺度 O(√d) 优于 Laplace 的 O(d)
- d=128 时噪声降低约 11 倍

隐私预算追踪采用 zCDP（zero-Concentrated DP）框架：
- Gaussian 操作消耗 ρ = Δ²/(2σ²)，ρ 线性可加
- 通过 ρ → (ε, δ) 转换公式回推等效消耗
- 同时维护 ε / δ / ρ 三个 `std::atomic<float>` 计数器，支持混合机制场景

### 2.5 联邦学习 — FederatedLearner

借鉴 FedProx 处理端侧计算性能差异，提供隐私保护的个性化情感模型服务。

FedAvg 联邦平均聚合算法：

```
w_global = Σ(n_k / N) · clip(w_k, C) + N(0, σ²)
```

| 符号 | 含义 |
|------|------|
| `n_k` | 节点 k 的样本数 |
| `N` | 总样本数 |
| `C` | L2 梯度裁剪阈值（防止单节点过度影响全局模型） |
| `σ` | DP 高斯噪声标准差（保护隐私） |

支持 FedProx 近端正则化（`mu` 参数），约束本地模型不偏离全局模型过远。每轮聚合后清空本地模型缓冲区，轮次计数器递增。

### 2.6 情绪脉搏 — EmotionPulseDetector

基于滑动窗口的实时情绪统计：

- 时间衰减加权：指数衰减函数，越新的样本权重越高
- 置信度加权：低置信度样本降权
- EWMA 平滑趋势斜率
- 鲁棒抗噪：中位数 + MAD（Median Absolute Deviation）离群点检测

输出 `EmotionPulse` 快照：加权平均分、标准差、情绪分布、趋势斜率、主导情绪。

### 2.7 模型量化 — ModelQuantizer

基于整数对称量化实现 float32 → INT8 推理：

- `quantizeToInt8`：计算 `scale = max(|tensor|) / 127`，`zeroPoint = 0`
- `quantizedMatMul`：INT8 矩阵乘法，手动 4 路循环展开优化
- `quantizedForward`：量化输入 → 矩阵乘 → 加偏置 → ReLU 激活

### 2.8 节点监控 — EdgeNodeMonitor

边缘节点健康监控：节点注册、状态上报（CPU / 内存 / 延迟）、负载均衡选择最优节点。

## 3. 情绪共鸣推荐

### 3.1 EmotionResonanceEngine

四维共鸣评分算法：

```
ResonanceScore = α·SemanticSim + β·TrajectoryDTW + γ·TemporalDecay + δ·DiversityBonus
```

默认权重及设计依据：

| 维度 | 权重 | 依据 |
|------|------|------|
| β 情绪轨迹匹配 | 0.35 | 核心差异化特性，相似情绪旅程的用户互动率最高 |
| α 语义相似度 | 0.30 | 保证内容相关性 |
| γ 时间衰减 | 0.20 | 避免推荐过旧内容 |
| δ 多样性奖励 | 0.15 | 防止回音室效应 |

DTW 优化：
- LB_Keogh 下界：O(n) 快速下界估计，用于剪枝
- LB_Improved 双向包络下界：比 LB_Keogh 更紧，剪枝率提升 20-40%
- Early Abandoning DTW：部分累积超过 best-so-far 时提前终止

权重自适应：
- EMA（指数移动平均）在线学习，根据用户隐式反馈调整四维权重
- `atomic<shared_ptr<const ResonanceWeights>>` 无锁一致读取，杜绝 torn-read

### 3.2 RecommendationEngine

五种推荐策略融合：

| 策略 | 方法 |
|------|------|
| User-CF | 基于潜在因子余弦相似度的用户协同过滤 |
| Item-CF | 基于共现矩阵的物品协同过滤 |
| 内容推荐 | 情绪向量兼容性匹配（`emotionCompatibilityScore`） |
| UCB 探索 | Upper Confidence Bound 多臂老虎机，平衡探索与利用 |
| 图传播 | 简化 PageRank 在用户-物品二部图上的传播 |

在线学习机制：
- SGD 增量更新潜在因子（`updateLatentFactors`）
- Thompson Sampling 自适应探索率
- 时间衰减权重：指数半衰期 24 小时

MMR（Maximal Marginal Relevance）多样性重排序，避免推荐结果过于同质。

### 3.3 AdvancedEmbeddingEngine

纯 C++ 实现的 128 维文本嵌入引擎，四层特征提取：

| 特征类型 | 维度占比 | 方法 |
|----------|----------|------|
| TF-IDF | 60% | 签名特征哈希降维，IDF 权重可增量学习 |
| 情感特征 | 15 维 | 中英文情感词典，提取极性 / 强度 / 波动 / 复杂度 |
| 统计特征 | 15 维 | 词汇多样性（TTR）、词频熵、词长分布 |
| N-gram | 10% | 2-gram / 3-gram 签名哈希，捕捉局部语序 |

性能优化：
- LRU 缓存（默认 10000 条）
- 4 路循环展开 + double 累加减少浮点误差
- N-gram 内联滑窗哈希，零中间 string 分配
- `std::jthread` 后台异步预热训练语料

## 4. 语义缓存

`SemanticCache` 实现两级缓存架构：

- L1 精确匹配：SHA256 哈希 → `unordered_map`，O(1)
- L2 语义匹配：查询 embedding 插入 `HNSWIndex`，O(log n) ANN 检索

当最近邻相似度超过阈值（默认 0.92）时视为语义命中，复用已缓存的响应。

淘汰策略：LFU（最少使用频率优先）+ TTL 过期清理（默认 86400 秒）。

预期效果：缓存命中率从 30% 提升至 60-70%，减少 40-50% AI API 调用。

## 5. 双记忆 RAG

`DualMemoryRAG` 双记忆架构：

短期记忆（`ShortTermEntry`）：
- 保留最近 5 次交互上下文
- 记录内容、情绪、分数、时间戳、访问次数
- 基于相关性淘汰（`evictLeastRelevant`），而非简单 FIFO

长期记忆（`LongTermProfile`）：
- 平均情绪分、指数衰减加权分（Ebbinghaus 遗忘曲线，λ=0.05）
- 主导情绪、情绪波动度、情绪趋势（rising / falling / stable）
- 连续负面天数（用于风险预警）

隐私保护：所有记忆使用 shadow_id，不关联真实身份。

`buildRAGPrompt` 将双记忆上下文注入 AI 提示词，生成个性化的情感回复。

## 6. 安全体系

### 6.1 PASETO v4 认证

替代 JWT 的无状态令牌方案，从根本上避免 JWT 的算法混淆攻击（`alg: none`）。`SecurityAuditFilter` 在请求到达 Controller 前统一校验，提取 `user_id` 注入请求属性。

管理后台使用独立的 `AdminAuthFilter` 和独立签名密钥（`ADMIN_PASETO_KEY`），与用户端完全隔离。

### 6.2 E2E 端到端加密

`E2EEncryption` 实现 IND-CCA2 安全性：

| 环节 | 算法 |
|------|------|
| 密钥交换 | X25519 ECDH（`computeSharedSecret`） |
| 密钥派生 | HKDF-SHA256（`deriveSessionKey`） |
| 认证加密 | AES-256-GCM（96-bit IV, 128-bit tag） |
| 前向保密 | 临时密钥对，每次会话独立 |

### 6.3 内容安全过滤

`ContentFilter` 三级过滤架构：

1. `CountingBloomFilter` 预检测：快速排除明显安全的文本，减少后续计算开销
2. `ACAutomaton` 精确匹配：Aho-Corasick 多模式匹配，O(n) 复杂度
3. `ShardedLRUCache` 结果缓存：热点文本直接命中

支持运行时热更新（`reload`），词条按类别和级别分级，白名单机制减少误报。文本预处理（`normalize`）统一全半角、移除零宽字符等干扰。

### 6.4 熔断器

`CircuitBreaker` 三态状态机：

```
CLOSED ──(连续失败达阈值)──→ OPEN ──(冷却超时)──→ HALF_OPEN
   ↑                                                    │
   └──────────(试探成功)────────────────────────────────┘
                                    │
                              (试探失败)
                                    ↓
                                  OPEN
```

`execute()` 采用两阶段锁策略：持锁检查状态（Phase 1），释放锁后执行业务逻辑（Phase 2），避免长时间持锁阻塞其他调用者。

## 7. 心理关怀体系

### 7.1 心理风险评估

`PsychologicalRiskAssessment` 融合两个维度：

语言学分析（`analyzeLinguisticMarkers`）：
- `detectSelfHarmIntent`：自伤意图关键词检测
- `detectHopelessness`：绝望感表达检测
- `detectIsolation`：社交孤立感检测

- 参与度下降趋势
- 活跃时段异常
- 社交孤立度

### 7.3 Guardian 守护者

守护者系统允许信任的人关注用户情绪状态。`GuardianIncentiveService` 提供激励机制，鼓励社区互助。

## 8. 高性能工程实践

### 8.1 并发安全

| 机制 | 应用场景 |
|------|----------|
| `std::shared_mutex` | 读多写少场景（缓存、HNSW 索引、情绪脉搏） |
| `std::atomic` | 无锁计数器（隐私预算、统计指标、启用标志） |
| `std::atomic<shared_ptr>` | 无锁一致读取（共鸣权重） |
| `std::once_flag` | 单例初始化 |
| `thread_local` | HNSW visited 标记，避免共享锁下的数据竞争 |

### 8.2 内存管理

- `std::unique_ptr` 管理子系统生命周期
- 空删除器包装静态单例，避免 DI 容器 double-free
- LRU 缓存容量限制 + TTL 过期，防止内存无限增长

### 8.3 算法优化

- 4 路循环展开：向量距离计算、归一化、余弦相似度
- ADSampling 提前终止：距离计算部分和超过阈值时返回 +inf
- 特征哈希降维：避免高维稀疏向量的内存开销
- N-gram 内联滑窗哈希：零中间 string 分配

Docker Compose 编排 6 个服务：

|------|------|------|------|
