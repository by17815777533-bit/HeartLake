# 测试验证与压测手册

> 本文档详述 HeartLake 的测试资产、运行方式、压测方案及性能指标目标。统一验收入口为 `./scripts/verify-2c2g.sh`，默认按 `2C2G` 资源标准执行：后端 `Release` 构建、构建/测试并发不超过 `2`、管理端 Node 内存上限 `512MB`、Flutter 测试并发 `2`。

当前项目的验收原则是“生产代码路径优先”。测试文件用于回归保护，但不作为唯一真相；涉及跨端契约、Provider 依赖反转、管理端运营面板和分页协议这类结构性改造时，必须同时看构建结果、手动冒烟与真实接口返回。

最新一次后端工程验证（2026-03）：
- `cmake --build backend/build-2c2g --target HeartLake -j 2` 通过
- 本轮重点验证的是算法与热链生产代码：`DualMemoryRAG`、`EmotionResonanceEngine`、`RecommendationController`、`HNSWIndex`、`EdgeAIEngine`、`VectorSearchController`、`StoneApplicationService`
- 后续补充验证还覆盖了 `InteractionApplicationService`、`InteractionController`、`PaperBoatController`、`UserController`
- 结论：推荐链、向量检索链、石头缓存链和边缘 AI 运行时配置链都已进入可编译状态，不能再用“测试文件质量不高”掩盖生产代码结构问题
- 2026-03 热路径收口后，建议至少追加执行 `cmake --build backend/build-2c2g --target HeartLake test_dual_memory_rag test_resonance test_edge_ai -j 2`，覆盖 RAG、共鸣推荐和边缘 AI 默认配置链

## 2. 后端测试

### 2.1 构建与运行

```bash
cd backend
cmake -S . -B build-2c2g -DCMAKE_BUILD_TYPE=Release
cmake --build build-2c2g -j2
ctest --test-dir build-2c2g --output-on-failure -j2
```

#### ONNX 模式构建

启用 ONNX Runtime 加载 BERT 情感模型（`sentiment_zh.onnx`）：

```bash
./scripts/restore-onnx-model.sh

cd backend
cmake -S . -B build-onnx \
  -DCMAKE_BUILD_TYPE=Release \
  -DHEARTLAKE_USE_ONNX=ON \
  -DOnnxRuntime_ROOT=$(pwd)/../third_party/onnxruntime-linux-x64-1.22.0 ..
cmake --build build-onnx -j2
LD_LIBRARY_PATH=../third_party/onnxruntime-linux-x64-1.22.0/lib:$LD_LIBRARY_PATH \
  ctest --test-dir build-onnx --output-on-failure -j2
```

说明：

- `benchmark_edge_ai` 中依赖公开情感数据集的测试项在数据集缺失时会自动 `SKIP`，不再把可选数据资源缺失误判为项目失败。
- ONNX 模型文件默认不提交 Git，统一通过 `./scripts/restore-onnx-model.sh` 恢复。
- 恢复模型后可执行 `VERIFY_ONNX_SMOKE=1 ./scripts/verify-onnx-smoke.sh`，验证模型、词表、Runtime 与基准链路全部可用。

### 3.2 测试分类

移动端测试策略以“直接覆盖生产代码路径”为原则。对于不调用真实生产逻辑、仅在测试文件内部复制响应处理器的低价值测试，已统一删除并改为共享契约测试，避免出现“测试通过但生产代码未被验证”的假阳性。

当前阶段的优先级是先完成生产代码的依赖反转、集合契约收敛和分页缓存治理，再补齐对应 Provider 测试。也就是说，测试设计要跟随稳定后的生产结构，而不是在旧架构上继续堆叠低价值用例。

最近几轮的移动端验收重点不再是“页面快照是否通过”，而是以下生产行为是否成立：

- 集合接口在 `semantic key / items / list / nested data` 间切换时，页面与 Provider 仍能稳定读取
- Provider 只依赖抽象数据源与实时通道接口，不再直接耦合单例实现
- 互动列表、石头详情、通知和好友链路的本地状态更新以后端成功结果为准，而不是测试里伪造的同步逻辑

## 7. 性能实测数据

以下全部数据来自本机实际运行结果，测试环境为 Linux 6.6（WSL2）。

### 7.1 情感分析精度（`benchmark_edge_ai` — SentimentAccuracyBenchmark）

#### 7.1.1 词典融合模式

50 样本中英文混合测试集，三层融合分析（规则 + 词典 + 统计）：

| 指标 | 实测值 |
|------|--------|
| 极性准确率（正/负/中性） | 94.0% |
| 情绪六分类准确率 | 88.0% |
| 宏平均 Precision | 92.6% |
| 宏平均 Recall | 90.0% |
| 宏平均 F1 | 90.3% |

各情绪类别明细：

| 情绪 | Precision | Recall | F1 | TP | FP | FN |
|------|-----------|--------|-------|----|----|-----|
| anger | 1.000 | 1.000 | 1.000 | 8 | 0 | 0 |
| surprise | 1.000 | 1.000 | 1.000 | 4 | 0 | 0 |
| sadness | 1.000 | 0.900 | 0.947 | 9 | 0 | 1 |
| fear | 1.000 | 0.833 | 0.909 | 5 | 0 | 1 |
| neutral | 0.667 | 1.000 | 0.800 | 10 | 5 | 0 |
| joy | 0.889 | 0.667 | 0.762 | 8 | 1 | 4 |

分析：anger / surprise 达到完美分类；sadness / fear 的 Recall 略低（各漏 1 例）；joy 的 Recall 最低（0.667），4 例 joy 被误分为 neutral；neutral 的 Precision 最低（0.667），5 例其他情绪被误判为 neutral。整体 F1 = 0.903，满足端侧轻量分析的精度要求。

#### 7.1.2 ONNX 模型模式

同一 50 样本测试集，启用 `-DHEARTLAKE_USE_ONNX=ON`，加载 `sentiment_zh.onnx`（BERT-base 中文，vocab 21128 tokens，CPU 推理，2 线程，SessionPool=1）：

| 指标 | 实测值 |
|------|--------|
| 极性准确率（正/负/中性） | 88.0% |
| 情绪六分类准确率 | 88.0% |
| 宏平均 Precision | 89.5% |
| 宏平均 Recall | 88.9% |
| 宏平均 F1 | 88.5% |

各情绪类别明细：

| 情绪 | Precision | Recall | F1 | TP | FP | FN |
|------|-----------|--------|-------|----|----|-----|
| anger | 1.000 | 1.000 | 1.000 | 8 | 0 | 0 |
| surprise | 1.000 | 1.000 | 1.000 | 4 | 0 | 0 |
| sadness | 1.000 | 0.900 | 0.947 | 9 | 0 | 1 |
| fear | 0.714 | 0.833 | 0.769 | 5 | 2 | 1 |
| joy | 0.800 | 1.000 | 0.889 | 12 | 3 | 0 |
| neutral | 0.857 | 0.600 | 0.706 | 6 | 1 | 4 |

分析：anger / surprise 完美分类；joy 全部召回（Recall = 1.000），F1 = 0.889；fear 有 2 例误报导致 Precision 降至 0.714；neutral 漏报 4 例（Recall = 0.600）。整体 F1 = 0.885，BERT 模型对隐含情绪的捕捉更到位，后续考虑结合词典模式做双通道融合投票。

### 7.3 推理延迟（`benchmark_edge_ai` — InferenceLatencyBenchmark）

#### 词典融合模式（Release 构建）

1000 次迭代取平均，单次操作延迟：

| 操作 | 延迟 | 说明 |
|------|------|------|
| `analyzeSentimentLocal` | 0.20 μs | 情感分析（三层融合） |
| `moderateTextLocal` | 0.15 μs | 内容审核（AC 自动机 + 风险评估） |
| `hnswSearch`（k=10） | 3.80 μs | HNSW 近似最近邻搜索 |
| `quantizeToInt8`（256 维） | 0.33 μs | float32 → INT8 量化 |

#### ONNX 模型模式（Debug 构建）

启用 ONNX Runtime（CPU，2 线程，SessionPool=1），1000 次迭代取平均：

| 操作 | 延迟 | 说明 |
|------|------|------|
| `analyzeSentimentLocal` | 32.92 μs | ONNX BERT 情感推理（含 tokenize） |
| `moderateTextLocal` | 0.75 μs | 内容审核（AC 自动机，不走 ONNX） |
| `hnswSearch`（k=10） | 43.54 μs | HNSW 近似最近邻搜索 |
| `quantizeToInt8`（256 维） | 1.59 μs | float32 → INT8 量化 |

## 8. 故障排查速查

| 症状 | 可能原因 | 排查方法 |
|------|----------|----------|
| 后端测试编译失败 | 缺少依赖库 | 检查 CMake 输出，安装缺失的 `-dev` 包 |
| ctest 部分失败 | 数据库未启动 | 确认 PostgreSQL / Redis 可达 |
| flutter test 失败 | 依赖未安装 | `flutter pub get` |
| vitest 失败 | node_modules 过期 | `cd admin && npm install` |
| ASAN 报错 | 内存安全问题 | 根据 ASAN 输出定位源码行号 |
| 压力测试崩溃 | 并发安全问题 | 检查 mutex / atomic 使用，用 TSAN 辅助定位 |
| docker-test 超时 | 容器启动慢 | 增加健康检查等待时间，检查资源限制 |
| ONNX 构建找不到头文件 | `OnnxRuntime_ROOT` 路径错误 | 确认 `third_party/onnxruntime-linux-x64-1.22.0/` 存在且包含 `include/` 和 `lib/` |
| ONNX 测试运行时 so 找不到 | `LD_LIBRARY_PATH` 未设置 | 运行前 `export LD_LIBRARY_PATH=.../onnxruntime-linux-x64-1.22.0/lib:$LD_LIBRARY_PATH` |
| ONNX 情感分析结果全为 neutral | 模型文件损坏或 vocab 不匹配 | 检查 `models/sentiment_zh.onnx` 和 `models/vocab.txt` 完整性 |
| DP 基准大量 "budget exhausted" 警告 | 隐私预算耗尽（预期行为） | 10,000 次采样会快速耗尽 ε=10 的预算，fallback 返回原值，不影响测试结果 |
