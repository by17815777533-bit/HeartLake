# 07 编码与注释规范（评委验收专用版）

本文档为 HeartLake 核心服务体系代码工程的最终规范蓝图，涵盖 C++（后端）、TypeScript（Admin 后台管理） 及 Dart（App 客服端） 的联合编码和注释工程指导，旨在保持全链路技术资产的高度易读性，满足架构级交付审查与技术文档闭环的要求。

## 一、 注释格式核心基准指南：Doxygen 泛化实现

全方案中各层级文件一律要求贯彻工业级 Doxygen 注释标准体系，具体要求：
- **拒绝生成式行文（即所谓的 “AI 味”）：** 使用标准计算机科学及软件工程官方业务术语和领域驱动架构（DDD）通用描述，不使用模糊的形容词或过渡口语化修辞。
- **信息遮蔽原则：** 注释内容严禁出现任何与特定开发者相关的名号（Author/Owner）、特定修订日期（Date/Timeline）、以及极易发生位移与更迭的文件实体名等（Filename），仅保留不可替代的技术语义和架构说明。

### 1.1 文件级声明区 (File Level Header)
部署在每一份具有独立领域业务代码的源代码物理文件顶部。

**模板（以 C++/TS 结构为例）：**
```cpp
/**
 * @brief <简要描述当前文件的核心领域划分及意图，通常为一至两句话长>
 *
 * <空行>
 * <详细描述具体的类、函数如何合作，存在什么架构依赖，以及主要的并发、异步、锁同步约定或内存回收范式说明>
 * <如存在网络IO和算法选型时说明具体策略：如指数退避（Exponential Backoff）、滑动窗口机制等>
 */
```

### 1.2 接口级/方法级声明区 (Method / Function Block)
应用于暴露对外的方法或存在大量数据交换（IO）、复杂计算处理的方法上方。

**模板：**
```cpp
/**
 * @brief <指令或函数意图说明>
 *
 * <对于复杂的生命周期函数或其他特殊内部处理，可追加一至二段架构或参数防注入的安全说明>
 *
 * @param params_name <参数意图定义与类型约定，避免直白翻译参数名>
 * @param config      <对配置数据结构的依赖和约束>
 * @return <对于方法的回溯结果及成功/失败信号位的表征说明>
 * @note <部分可能引起数据竞争或需要防死锁、避免竞态条件的额外警告及防范措施设计说明>
 */
```

### 1.3 核心依赖配置及行内定义 (Inline Declarations)
关键上下文常量定义、内存或容器指针结构旁，应辅以短注释：
```cpp
/** @brief 独立在子事件循环运行，规避 IO/CPU 死锁争夺或超时重定向干扰 */
```

## 二、 后端 C++ (Drogon/Milvus) 落标规范
所有服务端类结构严禁裸露参数块；在与向量数据库（Milvus）及 PostgreSQL 实体挂钩的持久化操作层，在对应的 Doxygen `brief` 下方说明具体安全转义策略。（例如：“通过清理或转义注入非法字符对后端分析解释器进行保护”）。所有方法头均需包含完整的 `@param` 与 `@return`。

## 三、 后台体系 Admin/Frontend (Vue/TS/Dart) 标引规范
- **Pinia Stores 及全局状态字典：** 全面增加针对于并发竞态以及持久化存储方案（LocalStorage/SessionStorage）降级保护与边界条件拦截说明。
- **Axios 与 HTTP 网络通信库：** 需要用严谨架构语体标记并发去重拦截装置（AbortController）、自动重试探测（Exponential Retry）及认证退避重定策略。
