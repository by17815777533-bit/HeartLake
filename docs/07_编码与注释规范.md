# 编码与注释规范

> 本文档定义 HeartLake 项目的编码与注释工程规范，涵盖 C++（后端）、TypeScript（管理后台）及 Dart（移动客户端）三端。旨在保持全链路代码资产的一致性与可读性，满足架构级交付审查要求。

## 1. 注释标准：Doxygen 规范

全项目统一采用 Doxygen 注释标准，具体要求：

- 使用标准计算机科学及软件工程术语，避免口语化表述和模糊修辞
- 注释中不包含开发者姓名（Author）、修订日期（Date）、文件名（Filename）等易变信息，仅保留技术语义和架构说明

### 1.1 文件级注释（File Level Header）

每个具有独立业务逻辑的源文件顶部须包含文件级注释：

```cpp
/**
 * @brief <文件核心职责的简要描述，一至两句话>
 *
 * <详细描述类 / 函数的协作关系、架构依赖、并发控制策略、
 *  内存管理范式等。如涉及网络 IO 或算法选型，说明具体策略
 *  （如指数退避、滑动窗口等）>
 */
```

### 1.2 方法级注释（Method / Function Block）

对外暴露的方法、涉及复杂计算或大量 IO 的方法须包含完整注释：

```cpp
/**
 * @brief <方法意图说明>
 *
 * <复杂生命周期函数或特殊内部处理的架构说明>
 *
 * @param paramName <参数语义与类型约束>
 * @param config    <配置数据结构的依赖和约束>
 * @return <返回值语义及成功 / 失败信号说明>
 * @note <数据竞争风险、死锁防范、竞态条件等注意事项>
 */
```

### 1.3 行内注释（Inline Declarations）

关键常量定义、内存 / 容器指针结构旁须辅以短注释：

```cpp
/** @brief 独立事件循环运行，规避 IO/CPU 锁争用 */
```

## 2. 后端 C++（Drogon）编码规范

### 2.1 架构约束

- 严格遵循 DDD 四层架构：Interfaces → Application → Domain → Infrastructure
- 头文件与实现分离：`.h` 置于 `include/`，`.cpp` 置于 `src/`
- 依赖方向：上层依赖下层接口，禁止反向依赖

### 2.2 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 类名 | PascalCase | `SentimentAnalyzer` |
| 方法名 | camelCase | `analyzeSentiment()` |
| 成员变量 | camelCase + 下划线后缀 | `cacheSize_` |
| 常量 | UPPER_SNAKE_CASE | `MAX_CACHE_SIZE` |
| 命名空间 | snake_case | `heartlake::ai` |

### 2.3 内存与并发

- 使用智能指针管理内存，禁止裸 `new` / `delete`
- 读多写少场景使用 `std::shared_mutex`
- 无锁计数器使用 `std::atomic`
- 单例初始化使用 `std::once_flag`
- 异常处理使用 `try-catch`，禁止未捕获异常逃逸

### 2.4 数据库操作

- 持久化操作层须在 Doxygen `@brief` 下方说明 SQL 注入防护策略
- 所有方法头须包含完整的 `@param` 与 `@return`
- 参数绑定使用预编译语句，禁止字符串拼接 SQL

## 3. 管理后台 TypeScript（Vue 3）编码规范

### 3.1 组件规范

- 使用 `<script setup>` 组合式 API
- 事件处理函数命名 `handleXxx`，数据获取函数命名 `fetchXxx`
- 危险操作（删除、封禁）须有二次确认弹窗

### 3.2 状态管理

- Pinia Store 须注释并发竞态处理策略
- 持久化存储（LocalStorage / SessionStorage）须说明降级保护与边界条件处理

### 3.3 网络通信

- HTTP 客户端须注释以下机制：
  - 并发去重拦截（AbortController）
  - 自动重试策略（指数退避）
  - 认证失效重定向策略
- 统一使用 `errorHelper` 处理错误

### 3.4 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 组件文件 | PascalCase | `UserList.vue` |
| 组合函数 | camelCase + use 前缀 | `useDashboardData` |
| Store | camelCase + use 前缀 + Store 后缀 | `useAuthStore` |
| 常量 | UPPER_SNAKE_CASE | `API_BASE_URL` |

## 4. 移动端 Dart（Flutter）编码规范

### 4.1 架构约束

- 遵循 DDD 分层：`data/datasources` → `domain/entities` → `presentation`
- 使用 Provider 进行状态管理
- 网络请求通过 Service 层封装，Screen 不直接调用 `ApiClient`
- 使用 `ServiceResponse<T>` 统一封装 API 响应

### 4.2 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | snake_case | `emotion_calendar_screen.dart` |
| 类名 | PascalCase | `EmotionCalendarScreen` |
| 方法名 | camelCase | `fetchEmotionData()` |
| 常量 | lowerCamelCase 或 UPPER_SNAKE_CASE | `defaultPageSize` |

### 4.3 注释要求

- 公开 API 使用 `///` 文档注释
- 复杂业务逻辑须说明算法选择和边界条件处理
- Provider 须注释状态更新时机和副作用

## 5. 通用规范

- 禁止在注释中包含开发者个人信息
- 禁止在代码中硬编码密钥或密码
- 禁止使用已废弃的 API
- 禁止提交未格式化的代码
