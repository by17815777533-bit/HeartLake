# HeartLake 架构修补与性能治理方案

## 1. 目标与边界

本轮治理优先覆盖 `backend` 与 `admin` 契约主链，Flutter 界面保持不动，并尽量保持既有管理端交互表现一致。治理目标不是局部补丁，而是一次性收敛以下问题：

- 字段命名和返回结构不一致
- 数据库表结构与代码实现脱节
- 旧表/旧字段/旧逻辑残留，难以判断是否可删
- 热路径查询、分页、列表刷新存在明显性能隐患
- 同类逻辑重复实现，维护成本高
- 石头互动写链依赖 `invalidatePattern("stone_list:*")`，热点期会触发 L1 缓存全量扫描和锁竞争

治理原则：

1. 先定标准，再批量修复
2. 入参兼容历史别名，出参收敛为标准字段
3. 不直接删高风险表，先归类、去引用、再清退
4. 优先修主链路，再修外围
5. 优先做低风险高收益性能优化
6. 热列表缓存优先使用作用域 namespace 失效，按 `owner + mood` 精准淘汰，避免前缀扫描式批量删除

## 2. 字段契约标准

### 2.1 用户对象标准字段

标准用户对象统一输出以下字段：

| 标准字段 | 兼容输入/历史别名 | 说明 |
|---|---|---|
| `user_id` | `id`, `userId` | 业务用户主标识 |
| `username` | - | 用户名 |
| `nickname` | - | 昵称 |
| `avatar_url` | `avatar`, `avatarUrl`, `avatar_path`, `image` | 头像地址 |
| `bio` | `signature` | 个人简介 |
| `gender` | - | 性别 |
| `birthday` | - | 生日，`YYYY-MM-DD` |
| `location` | - | 地区 |
| `email` | - | 邮箱 |
| `status` | - | 账号状态 |
| `created_at` | `createdAt` | 创建时间 |
| `last_active_at` | `lastActiveAt` | 最后活跃时间 |

约束：

- 前端内部继续兼容旧字段，但对外传输与核心数据层全部以标准字段为准。
- 后端返回体禁止新增 `avatar`、`userId`、`createdAt` 这类变体。

### 2.2 列表与分页标准字段

标准分页返回结构：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20
}
```

兼容读取时允许识别：

- 列表：`items`, `list`, `results`, `data`, `users`
- 总数：`total`, `count`, `total_count`, `pagination.total`, `meta.total`

但服务层归一后，向上层统一暴露：

- `items`
- `total`

如业务语义更明确，可额外镜像出：

- `users`
- `stones`
- `blocked_users`

### 2.3 上传结果标准字段

标准上传结果统一为：

```json
{
  "url": "https://..."
}
```

兼容读取：

- `file_url`
- `avatar_url`
- `media_url`

### 2.4 连接消息标准字段

标准消息对象：

| 标准字段 | 说明 |
|---|---|
| `message_id` | 对外消息标识，允许由数据库数值 `id` 转为字符串暴露 |
| `connection_id` | 所属连接 |
| `sender_id` | 发送者 |
| `content` | 内容 |
| `created_at` | 发送时间 |

说明：

- 数据库层 `connection_messages` 当前真实主键是 `id SERIAL`。
- 应用层对外仍可保留 `message_id` 字段，避免前端大面积改动。

## 4. 已确认的结构性问题

### 4.1 用户主键双轨问题

当前项目同时存在：

- `users.id`
- `users.user_id`

历史迁移与业务代码混用这两套字段，导致：

- 表外键指向不统一
- 查询 / 删除 / 关联经常写错字段
- 新功能很容易继续复制历史错误

当前修补策略：

- 业务接口一律以 `user_id` 为外部用户标识
- 数据库层保持 `users.id == users.user_id`
- 通过迁移和触发器阻止两列再次漂移
- 核心业务表外键统一迁移到 `users(user_id)`

### 4.2 账户模块与真实 schema 脱节

主要表现：

- `AccountController` 使用了不存在的表
- `deleteAccountPermanently()` 删除了大量不存在或列名错误的对象
- 用户资料接口读取了 schema 中未稳定补齐的字段
- 拉黑接口依赖 `user_blocks`，但迁移未创建

### 4.3 连接消息模块列名错误

主要表现：

- `connections` 真实列为 `user_id / target_user_id`
- 代码仍在使用 `user_id_1 / user_id_2`
- `connection_messages` 真实主键为 `id`
- 代码仍在写入 / 读取 `message_id`

### 4.4 EdgeAI 控制平面与启动链漂移

主要表现：

- 启动期曾把 `edge_ai_enabled` 硬编码为开启，后台配置关闭后仍会初始化 EdgeAI 主链
- admin 端保存的 `onnx_enabled / model_path / vocab_path / onnx_threads` 与后端启动配置脱节，容易出现“保存成功但重启后没生效”
- 管理端读取配置时同时依赖 root 字段和 `data.config`，而后端历史上只返回其中一部分字段

当前修补策略：

- EdgeAI 启动统一优先读取 `AdminConfigStore::load()["ai"]`，并以 `EDGE_AI_*` 环境变量作为默认值/兜底
- `EdgeAIController` 统一输出 snake_case + camelCase 镜像字段，同时保留 `config` 镜像，避免后台读写链漂移
- 运行时可热应用键和需要重启键明确分层，关闭状态下的 EdgeAI 子系统改为延迟初始化，减少 2c2g 机器启动期的无谓成本

## 5. 性能热点清单

### P0：必须优先处理

1. 后端大量 `SELECT *`
   - 代表文件：
     - `backend/src/domain/stone/repositories/StoneRepository.cpp`
     - `backend/src/domain/friend/repositories/FriendRepository.cpp`
   - 风险：
     - 无谓取数
     - schema 变更放大影响
     - 序列化负担增加

2. 热列表 `OFFSET` 分页
   - 代表场景：
     - 石头流
     - 互动消息
     - 通知 / 纸船列表
   - 风险：
     - 深分页越来越慢
     - 大数据量下排序代价高

3. WebSocket 事件触发整列表扫描与整页刷新
   - 代表文件：
     - `frontend/lib/presentation/screens/lake_screen.dart`
     - `frontend/lib/presentation/providers/stone_provider.dart`
   - 风险：
     - 高频事件下 UI 抖动
     - 列表规模增大后每次更新退化为 O(n)

## 6. 2026-03-21 后端热点优化落地

本次只调整 `backend/src/application`、`backend/src/interfaces/api`、`backend/include/application`，未触碰 Flutter 界面。

### 6.1 用户接口重复编排收敛

- `UserController::getUserInfo()` 改为复用 `UserApplicationService::getUserProfile()`，公开资料读取统一走服务层缓存与聚合 SQL。
- `UserController::searchUsers()` 改为复用 `UserApplicationService::searchUsers()`，控制器不再重复拼分页查询。
- `UserController::updateProfile()` 改为只做入参校验，动态 SQL 与缓存处理全部下沉到 `UserApplicationService::updateUserProfile()`。

收益：

- 用户公开资料查询命中应用层缓存，减少重复数据库访问。
- 用户搜索和资料更新的 SQL 绑定集中在服务层维护，减少控制器重复逻辑和后续漂移风险。

### 6.2 连接消息双查合并

- `InteractionApplicationService::getConnectionMessages()` 将“权限校验 + 分页查询”合并为单条 SQL。
- `InteractionApplicationService::createConnectionMessage()` 将“参与者校验 + INSERT”合并为单条 SQL。

收益：

- 连接消息列表和发消息路径各减少一次数据库往返。
- 保留原有越权语义：无访问权限时仍返回业务错误，不把空列表和越权混淆。

### 6.3 本轮未纳入

- `consultation / friend / notification / admin` 模块按本轮边界未修改。
- 深分页从 `OFFSET` 切换为游标分页仍是后续治理项，本次仅处理可低风险落地的双查与重复编排。

### 6.4 EdgeAI 启动与管理端契约收敛

- `main.cpp` 启动期不再硬编码 `enabled=true`，而是读取持久化 `ai` 配置中的 `edge_ai_enabled / onnx_enabled / model_path / vocab_path / onnx_threads / hnsw_* / dp_* / pulse_window_seconds / sentiment_cache_*`
- `EdgeAIController::getAdminConfig()` 统一返回 root 层字段、`config` 镜像、`engine_stats` 和 `engine_enabled`，后台读取不用再猜字段层级
- `EdgeAIController::updateAdminConfig()` 现在会把配置持久化到 `AdminConfigStore`，并区分 `applied_keys` 与 `restart_required_keys`
- `EdgeAIEngine` 在禁用态启动时改为延迟创建情感分析、内容审核、HNSW、联邦学习和节点监控对象；后续运行时启用时再补建，避免 2c2g 节点启动期空耗

1. 同步数据库调用过多
   - 部分主链路还在 `execSqlSync`
   - 高并发下容易占住工作线程

2. 统计查询分散
   - 多个接口重复 `COUNT(*)`
   - 后续可做聚合视图或汇总缓存

3. 页面内嵌 `shrinkWrap` 网格/列表
   - 代表文件：
     - `frontend/lib/presentation/screens/vip_screen.dart`
   - 风险：
     - 额外布局测量
     - 长页面掉帧

### P2：中期优化

1. Provider 粒度过粗，`notifyListeners()` 触发范围偏大
2. 多处 JSON 解析与字段归一逻辑重复
3. 局部列表缺少键控索引或增量更新容器

## 6. 执行顺序

### 第一批：结构止血

- 用户字段契约统一
- 账户 schema 对齐
- `user_blocks` 落库
- 账户删除逻辑改为只删除真实存在的对象
- 连接消息列名修正

### 第二批：热路径性能治理

- 去除主链路 `SELECT *`
- 为连接消息、账户拉黑等热点补索引
- 收敛同步数据库调用
- 热列表分页策略评估并逐步转向游标分页

### 第三批：前端列表与状态管理优化

- 石头流事件更新改为索引化 / 增量更新
- 降低高频 `setState` / `notifyListeners` 成本
- 清理嵌套滚动与高代价布局

### 第四批：遗留对象清退

- 删除零引用旧逻辑
- 下线待清退表
- 同步迁移、脚本、测试与文档

## 7. 第一批改造验收标准

### 正确性

- 账户相关接口不再因缺表缺列报错
- 拉黑 / 取消拉黑链路可用
- 数据导出链路可正常创建任务
- 永久删号不再访问不存在的表或错误列
- 连接消息发送 / 拉取恢复可用

### 性能

- 主链路热点仓储不再使用 `SELECT *`
- 新增索引覆盖 `user_blocks` 与连接消息时间排序

### 稳定性

- 管理端契约调整不改变既有页面交互路径
- Flutter 界面表现不变
- 兼容历史字段输入
