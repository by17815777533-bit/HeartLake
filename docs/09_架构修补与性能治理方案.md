# 架构修补与性能治理方案

本文描述 HeartLake 当前的工程约束和性能治理重点。

## 1. 当前架构约束

- 失败不允许伪装成空数据。
- 集合接口必须显式返回总数。
- 实时事件必须统一事件壳。
- 写链确认成功后才更新本地状态。
- AI、推荐、共鸣失败必须显式暴露。

## 2. 当前关键治理点

### 认证链

- PASETO 密钥按当前实现做进程内缓存
- token 校验路径只解析一次 JSON payload
- 管理员和用户 token 使用独立密钥

### 实时链

- `BroadcastWebSocketController` 的握手成功和错误消息保持最小 JSON
- 业务广播统一走标准事件构造
- WebSocket 房间消息受鉴权和房间权限保护

### 集合链

- 移动端、管理端、后端统一使用 `items / total / page / page_size`
- 页面和图表不再根据长度猜分页
- 查询失败不把旧结果伪装成新成功结果

### AI 与推荐链

- 高级推荐直接暴露真实算法结果
- 共鸣搜索直接暴露真实匹配结果
- 降级必须显式标注来源或 `degraded`

## 3. 当前性能治理点

- WebSocket fan-out 和握手长尾
- `stones`、`users`、`emotion_tracking`、`stone_embeddings` 顺序扫描
- 推荐和共鸣链长尾分位
- 管理端统计链的去重和 stale 控制

## 4. 当前运行风险

- PostgreSQL `collation version mismatch`
- 数据量进一步增大后的索引退化
- 向量检索和高级推荐在高峰时段的长尾抖动

## 5. 当前优化优先级

1. 数据库顺序扫描继续下降
2. WebSocket 房间广播成本继续下降
3. 高级推荐和共鸣搜索长尾继续压缩
4. 管理端统计链刷新成本继续下降
