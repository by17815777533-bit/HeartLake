# HeartLake 本地启动与运行手册

## 1. 目标
本手册用于在评审、演示、联调三种场景下，稳定启动 HeartLake 全链路环境。内容覆盖：
- 非 Docker 方式（便于开发调试、定位源码问题）
- Docker 方式（便于快速复现、交付验收）
- 启动顺序与每一步背后的技术原因

## 2. 运行架构与技术选型说明
- 后端：C++20 + Drogon，监听 `8080`。
  - 选择原因：高并发下延迟稳定，可直接控制线程与连接池，便于压测达标。
- 前端：Flutter Web（调试阶段使用 DDC）。
  - 选择原因：同一套业务逻辑可覆盖 Web/移动端，减少多端逻辑分叉。
- 数据库：PostgreSQL 16。
  - 选择原因：强事务一致性 + JSON 能力，适合社交内容与关系数据。
- 缓存：Redis 7。
  - 选择原因：高频读写（通知数、会话态、热数据）可降后端负载。
- 模型服务：Ollama（本地推理入口，默认 `11434`）。
  - 选择原因：本地可控、离线可跑、便于替换模型与压测稳定性。

## 3. 启动前检查
- 检查端口占用：
```bash
ss -ltnp | rg ':3000|:8080|:11434|:5432|:6379'
```
- 必需目录：
  - 后端源码：`backend/`
  - 前端源码：`frontend/`
  - 运行日志：`.runtime/logs/`

## 4. 非 Docker 启动（推荐开发态）

### 4.1 启动后端
```bash
cd /home/joker/code/heartlake/backend
./start.sh
```

技术说明：
- `start.sh` 会按迁移顺序补齐数据库结构，再构建并启动 `HeartLake`。
- 这样做的原因是保证“代码版本”和“数据库结构”一致，避免接口 500 由 schema 漏项触发。

健康检查：
```bash
curl -sS http://127.0.0.1:8080/api/health
```

### 4.2 启动前端
```bash
cd /home/joker/code/heartlake/frontend
flutter pub get
flutter run -d web-server --web-hostname 0.0.0.0 --web-port 3000
```

技术说明：
- `web-server` 方式便于局域网联调。
- 本项目在 `frontend/web/flutter_bootstrap.js` 中会在本地环境主动清理 Service Worker，避免旧缓存导致“页面不更新/白屏假象”。

访问地址：
- 前端：`http://127.0.0.1:3000`
- 后端：`http://127.0.0.1:8080`

### 4.3 启动 Ollama（若未运行）
```bash
ollama serve
```

检查：
```bash
curl -sS http://127.0.0.1:11434/api/tags
```

## 5. Docker 启动（交付验收态）
```bash
cd /home/joker/code/heartlake

```

技术说明：
- Docker 模式用于“快速起整套服务”，避免机器环境差异影响验收。
- 本地调试仍建议优先非 Docker，便于直接附加调试器和查看源码栈。

检查：
```bash
docker compose ps
curl -sS http://127.0.0.1:8080/api/health
```

## 6. 推荐启动顺序与原因
1. PostgreSQL / Redis
2. Ollama
3. 后端
4. 前端

原因：
- 后端启动时会初始化依赖与模型能力，若基础服务不可达会触发降级或启动失败。
- 前端最后启动可减少启动期无效请求与误报。

## 7. 常见问题与处理

### 7.1 前端白屏
排查顺序：
1. `3000` 是否在监听。
2. `8080` 是否在监听。
3. 浏览器控制台是否停在 `DDC is about to load ...`。
4. 强刷缓存 `Ctrl + Shift + R`。
5. 重启前端 `flutter run`。

### 7.2 接口 500
- 先看后端日志是否是 schema 缺失。
- 执行后端迁移后重启。

### 7.3 功能看起来“没生效”
- 优先确认 token 和 user_id 是否正确写入存储。
- 再确认请求是否命中旧缓存。

## 8. 关闭服务
- 前台进程：`Ctrl + C`
- 后台清理：
```bash
pkill -f '(^|/)HeartLake($| )' || true
pkill -f 'flutter_tools.snapshot run -d web-server' || true
pkill -f 'ollama serve' || true
```

## 9. 关键环境变量与来源说明
以下变量决定了“是否能稳定跑起来”，建议评审机提前核对：

- `PASETO_KEY`
  - 用途：服务端签发/校验匿名登录 token。
  - 原因：无状态鉴权必须有稳定密钥；密钥长度不足会在启动时被拒绝。
- `DB_HOST / DB_PORT / DB_NAME / DB_USER / DB_PASSWORD`
  - 用途：后端数据库连接池初始化。
  - 原因：社交和情绪链路都强依赖 PostgreSQL，启动通过但数据库不可写会导致运行期大量 500。
- `REDIS_HOST / REDIS_PORT`
  - 用途：热点缓存、实时状态、限流辅助。
  - 原因：Redis 不可用时虽然有回退逻辑，但高并发压测下延迟会明显上升。
- `AI_PROVIDER / AI_BASE_URL / AI_MODEL`
  - 用途：LLM 推理与 AI 对话入口。
  - 原因：模型服务不可达时会触发降级逻辑，功能可用性下降但系统仍可运行。
- `EDGE_AI_ONNX_FORCE_GPU`
  - 用途：是否强制 ONNX 走 GPU EP。
  - 原因：评审机无 CUDA runtime 时应设为 `false`，否则可能出现启动时 provider 动态库加载失败。

## 10. 启动链路中的关键技术点（评审常问）

### 10.1 为什么后端先跑迁移再起服务
- 现象：脚本每次会顺序执行 `backend/migrations/*.sql`。
- 技术原因：HeartLake 的业务接口覆盖面广，字段缺失会在运行期表现为随机 500；先迁移可把错误前置到启动阶段。

### 10.2 为什么 Web 端要主动处理 Service Worker 缓存
- 现象：Flutter Web 改动后，旧缓存会造成“代码已修但页面还旧”。
- 技术原因：评审阶段频繁迭代，缓存污染比代码错误更常见；清缓存策略能明显降低白屏和假回归问题。

### 10.3 为什么保留非 Docker 启动
- 现象：同一仓库提供非 Docker 与 Docker 两套运行方式。
- 技术原因：非 Docker 便于直接附加调试器和看源码栈；Docker 便于交付时快速复制环境。两者互补，不互斥。

## 12. 启动失败定位矩阵（按症状反查）
- 症状：`8080` 返回 `502`
  - 优先排查：网关还在但后端主进程未启动。
  - 操作：检查后端主进程是否存在，再看启动日志尾部首条 `ERROR/WARN`。
- 症状：`POST /auth/anonymous` 500
  - 优先排查：数据库 schema 与连接参数。
  - 操作：手动执行一条简单 SQL 验证读写权限。
- 症状：AI 对话总是降级回复
  - 优先排查：`AI_BASE_URL` 是否可达、模型是否已拉取。
  - 操作：直接 `curl` 模型服务健康接口。
- 症状：前端更新后仍是旧页面
  - 优先排查：浏览器缓存与 Service Worker。
  - 操作：强刷后重启 web server，再看 Network 面板是否命中新资源。

## 13. 关键技术点来历（评审可直接引用）

### 13.1 为什么后端是 C++ + Drogon
- 历史背景：项目初版使用脚本服务做原型，功能可跑但在并发读写时抖动明显。
- 迁移理由：
  - 社交场景请求短平快，线程池 + 连接池模型更容易控制尾延迟。
  - 核心链路需要可预测的资源开销（尤其压测阶段）。
- 实际收益：
  - 同机压测下，`p95` 延迟稳定在可接受范围，错误率可收敛到低位。

### 13.2 为什么前端采用 Flutter Web
- 历史背景：最早尝试双端分离（Web + 移动两套 UI），维护成本过高。
- 选型理由：
  - 复用业务模型和服务层，减少“同功能不同实现”导致的回归问题。
  - 对评审场景友好，一套代码可演示多端能力。

### 13.3 为什么 Redis 只做短 TTL 热缓存
- 历史背景：早期把过多业务状态长期放入缓存，导致失效链路复杂、错读概率高。
- 当前策略：
  - 仅缓存高频读接口，TTL 明确。
  - 写操作成功后主动失效前缀，优先一致性。
- 结论：缓存负责加速，不承担事实真相，数据库仍是权威源。

## 14. 启动时序中的依赖关系图（文字版）
1. 加载 `.env` 并导出运行时环境变量。  
2. 生成 `config.runtime.json`，作为 Drogon 实际读取配置。  
3. 检查 PostgreSQL 可用性，必要时初始化本地实例。  
4. 检查 Redis 可用性，必要时启动本地实例。  
5. 执行 SQL 迁移，确保 schema 与代码对齐。  
6. 构建后端二进制（可通过 `HEARTLAKE_SKIP_BUILD=true` 跳过）。  
7. 启动后端并做 `/api/health` 检查。  
8. 启动 Flutter Web，前端开始请求 API。  

为什么按这个顺序：
- 避免“服务先起来但依赖没准备好”的假健康状态。
- 把 schema 错误前置到启动阶段，减少运行期 500。

## 15. 运行时配置项的设计原因
- `DB_CONNECTION_POOL_SIZE`
  - 过小会形成连接争用；过大则放大数据库切换成本。
  - 本地开发建议 `12~20`，评审压测可适当上调。
- `REDIS_CONNECTION_POOL_SIZE`
  - 用于高频 GET 缓存和状态读写；过小会放大排队延迟。
- `SERVER_THREADS`
  - 与 CPU 核数和 IO 模型相关，盲目开大不会线性提速。
- `AI_OLLAMA_FORCE_GPU / EDGE_AI_ONNX_FORCE_GPU`
  - 仅在 GPU 运行时可开启；评审机无 CUDA 时必须关闭强制 GPU。

## 16. 常见误区
- 误区：`/api/health=ok` 就代表全链路可用  
  - 事实：健康检查只表示进程在线，不代表业务依赖都正常。
- 误区：出现白屏一定是前端代码错  
  - 事实：本地缓存、旧 Service Worker、后端 502 同样会造成白屏。
- 误区：压测时先开前端再开后端  
  - 事实：前端会先打大量失败请求，污染日志并影响问题定位。
