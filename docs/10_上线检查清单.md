# HeartLake 上线检查清单

> 这份清单面向“准备正式上线”的最后一公里。它只保留必须执行的动作，不讨论实现细节。

## 1. 配置确认

- `.env` 中数据库密码、Redis 密码、PASETO 密钥、管理员密码哈希都已替换为生产值。
- `AI_API_KEY`、`AI_BASE_URL`、`AI_MODEL` 已按生产环境填写，不使用占位值。
- 若走 `server-lite`，已确认后台扫描开关是否符合服务器资源上限。
- 网关对外地址、域名、反向代理路径已确认，三端统一通过网关访问。
- 管理员账号、运维 SSH、仓库推送权限都已确认，不依赖本地开发者缓存。

## 2. 构建与静态校验

- `./scripts/verify-2c2g.sh` 通过。
- `./scripts/docker-test.sh` 通过。
- 若本次涉及 ONNX / EdgeAI 模型改动，`VERIFY_ONNX_SMOKE=1 ./scripts/verify-onnx-smoke.sh` 通过。
- Git 工作树干净，待上线提交已推到远端分支。

## 3. 核心链路手动验收

至少手动走完一次以下链路：

1. 匿名登录 -> 投石 -> 湖面列表可见。
2. 涟漪创建 / 删除 -> 计数与实时广播一致。
3. 纸船发送 -> 石头详情、我的纸船、通知一致。
4. 临时好友列表 -> 私聊 -> 删除临时好友。
5. 好友列表 -> 私聊消息收发。
6. 个人资料 -> 隐私设置 -> 拉黑 / 取消拉黑。
7. 情绪日历 -> 热力图 -> 情绪脉搏 -> AI 情绪分析。
8. SafeHarbor / 守望者 / 灯火计划 / 咨询链至少各打开并请求一次。
9. 管理后台首页、用户页、内容页、举报页、系统配置页至少打开一次。

## 4. 当前必须认清的产品语义

上线前需要确认运营和前端都知道这些约束：

- 好友关系由亲密度自动判定，`accept/reject/pending` 已下线。
- 临时好友只由石头/纸船互动自动建立；手动直连和手动升级永久好友已下线。
- 账号停用和兼容删号路由必须显式提供 `confirmation`。
- 情绪日历 / 热力图 / 情绪脉搏不会再把坏载荷伪装成默认中性数据。
- Edge AI 管理配置对非法值直接拒绝，不会偷偷回退默认值。

## 5. 部署动作

低配 Ubuntu 服务器优先按这组命令：

```bash
./scripts/docker-up.sh server-lite
```

增量发布时只用需要的那条：

```bash
./scripts/docker-up.sh server-lite-admin
./scripts/docker-up.sh server-lite-backend
./scripts/docker-up.sh server-lite-gateway
```

如果是本地构建后推远端：

```bash
./scripts/deploy-server-lite-local.sh admin <remote-host>
./scripts/deploy-server-lite-local.sh backend <remote-host>
./scripts/deploy-server-lite-local.sh all <remote-host>
```

## 6. 上线后冒烟

上线后立即检查：

```bash
curl -fsS http://127.0.0.1/healthz
curl -fsS http://127.0.0.1/api/health
curl -I http://127.0.0.1/admin/
```

如果走外网域名，也要从外部网络再检查一次：

- `/healthz`
- `/api/health`
- `/admin/`
- `/ws/broadcast` 握手

## 7. 回滚准备

上线前就要准备回滚，不要等故障后再想：

- 记录当前线上提交号、镜像标签、`.env` 版本。
- 确认数据库迁移是否只前进、不支持快速回滚；如涉及结构变更，先备份。
- 若本次只改 admin/backend/gateway，优先用对应的增量重建命令回滚。
- 保留上一版可用镜像或上一版 Git 提交，避免现场临时找版本。

## 8. 上线完成标准

满足以下条件才算上线完成：

- 健康检查通过。
- 三端至少各完成一次真实访问。
- 管理后台可登录，可读到实时统计。
- 核心业务链无 500、无白屏、无“成功但空数据”的假状态。
- 运维知道当前发布提交号和回滚路径。
