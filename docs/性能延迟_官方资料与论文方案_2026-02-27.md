## 从资料提炼的落地结论
1. CPU-only 下要优先做线程池与会话池协同，而不是盲目堆并发。
2. 若能提供 CUDA EP 运行库，GPU 可显著改善高并发 tail latency；当前日志显示 CUDA provider 动态库缺失，已回退 CPU。
3. INT8 动态量化 + ORT format 是在不改业务接口前提下最稳妥的低风险提速路径。
4. 高并发下应采用“队列可控 + 微批处理”策略（Triton/服务内微批），避免随机排队放大 p95/p99。
5. 模型层建议保留你当前“双路策略”：本地快速分支 + 高精度分支按不确定度触发，而不是单路硬判。

## 建议优先级
1. P0：补齐 CUDA EP 运行库并验证 GPU 实跑（先把 CPU fallback 问题解决）。
2. P0：导出并切换 ORT format，做 INT8 动态量化 A/B。
3. P1：按并发目标做 ONNX 线程池/会话池栅格搜索（固定口径压测）。
4. P1：在推理入口引入可控微批（小批次+短等待上限），压 tail latency。
5. P2：继续优化双路门控阈值，确保精度收益不被性能策略抵消。
