## 二、架构改进（三项）

### 2.1 Admin 迁移 TypeScript

当前 Admin 整个项目是纯 JavaScript，API 响应无类型，到处防御性代码。

迁移方案：
- `vite.config.js` → `vite.config.ts`
- 所有 `.vue` 文件 `<script>` → `<script setup lang="ts">`
- 定义 API 响应类型接口（User、Stone、Dashboard 等）
- Store 加类型定义
- 逐步迁移，优先核心页面（Dashboard、Users、EdgeAI）

收益：编译期类型检查，减少运行时错误，IDE 智能提示。

### 2.2 Element Plus 按需导入

当前全量导入，打包体积偏大。

- 改用 `unplugin-vue-components` + `unplugin-auto-import`
- 自动按需导入 Element Plus 组件
- 预计减少 60%+ 的打包体积

### 2.3 Flutter 状态管理增强

当前仅 3 个 Provider（AuthProvider 等），大量页面自行管理状态，导致：
- 状态分散，跨页面数据同步困难
- 重复请求，无全局缓存

改进方案：
- 新增 StoneProvider — 石头列表、投石/捡石状态
- 新增 FriendProvider — 好友列表、临时好友
- 新增 NotificationProvider — 通知未读数、实时推送
- 新增 EdgeAIProvider — 情感分析结果、AI 推荐
- 各 Screen 从 Provider 读取状态，不再自行管理

## 三、熔断器模式

在 Flutter Service 层实现 Circuit Breaker：

```
Closed（正常）→ 连续失败 N 次 → Open（熔断，返回降级结果）
                                    ↓ 超时后
                               Half-Open（试探放行一个请求）
                                    ↓ 成功则恢复 Closed
```

适用场景：
- API 请求失败时自动降级为缓存数据
- 后端不可用时快速失败，避免用户长时间等待

## 四、链路追踪（OpenTelemetry）

给每个请求打 trace-id，追踪完整链路：

```
Flutter → Nginx → Drogon → EdgeAI / PostgreSQL / Redis → WebSocket
```

- 需要在 Drogon 中间件手动埋点（无现成插件）
- 可视化面板用 Jaeger 或 Zipkin
- 竞赛演示时展示请求链路，视觉冲击力强
